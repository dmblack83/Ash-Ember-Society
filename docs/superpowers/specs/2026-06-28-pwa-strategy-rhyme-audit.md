# PWA Strategy Audit: rhyme.com vs Ash & Ember

> Evaluation doc, not an implementation plan. Purpose: decide Ash & Ember's PWA
> direction by comparing it head-to-head with rhyme.com, a fast, well-behaved iOS
> PWA Dave admired. Written 2026-06-28.

---

## 1. The question this answers

Dave's strategic question: **PWA-first vs Web-first + mobile-responsive.**

Short answer from the evidence: that is the wrong axis. rhyme is **also** PWA-first
and mobile-first. It did not win by choosing a different strategy than us. It won by
choosing a far **simpler service-worker architecture** — and by refusing to take on
the one hard problem we took on: caching the logged-in app for offline use.

The real fork is: **offline-first app-shell caching, yes or no.** Almost everything
that has cost us reliability pain traces back to that one decision.

---

## 2. What rhyme actually does (ground truth from their code)

Pulled live from rhyme.com on 2026-06-28.

**Stack:** Next.js App Router + Turbopack, served via Cloudflare in front of an AWS
origin. The authenticated app lives under `/beta/*` and is gated; `/` is a public
marketing + login shell.

**Service worker (`/sw.js`) — 3 KB, hand-written. Full strategy:**
- Precaches exactly **3** assets on install: `/`, `/manifest.json`, one SVG icon.
- Network-first for everything else: fetch live, write a copy to cache, fall back to
  cache only when the network fails.
- **Explicitly does not cache the app.** Skips `/api/*`, `/beta/*`, `/login`,
  `/verify-email`, and all cross-origin. Their own comment: caching the authed shell
  would "either leak app shell to the wrong user OR pollute the cache with 404 HTML."
- Offline navigation fallback: serve cached `/`.
- That is the entire worker. No precache manifest, no per-resource strategy table, no
  offline-mutation replay, no auth-partitioned cache keys, no telemetry bridge.

**Manifest:** `display: fullscreen` (override chain fullscreen → standalone →
minimal-ui), one SVG icon at `sizes: "any"` for both `any` and `maskable`, portrait,
theme + background both `#F8F2E6`. `start_url` and `id` both `/?source=pwa` (PWA-launch
analytics tag).

**iOS head:** `viewport-fit=cover, user-scalable=no, maximum-scale=1`,
`apple-mobile-web-app-status-bar-style: black-translucent`, `mobile-web-app-capable`.
**No apple-touch-startup-image at all** — no splash screens. They rely on
`background_color` + fullscreen for the launch feel.

**Headers:** HSTS, X-Frame-Options, nosniff, referrer-policy, permissions-policy
locking camera/mic/geo. Static chunks `immutable, max-age=1yr`. **CSP is report-only**
(same posture we landed on after enforce broke prod).

## 3. What Ash & Ember does (ground truth from our code)

**Stack:** Next.js App Router on Vercel. Whole app is authenticated and PWA-installed.

**Service worker (`app/sw.ts` → `public/sw.js`) — Serwist, with 8 runtime-caching
buckets**, evaluated top-down:

| Resource | Strategy |
|---|---|
| Supabase REST/Auth | NetworkOnly |
| Supabase public storage | StaleWhileRevalidate (6d) |
| RSC payloads | NetworkOnly |
| `/_next/static/*` | CacheFirst (1yr) |
| `/_next/image` | StaleWhileRevalidate (30d) |
| Same-origin images | StaleWhileRevalidate (30d) |
| Google Fonts | CacheFirst (1yr) |
| **Navigation (the app shell)** | **StaleWhileRevalidate, auth-partitioned** |

Plus: precache manifest of `/_next/static` hashes, `/offline` fallback, push
notifications, **offline mutation replay** (IndexedDB outbox + Background Sync),
`skipWaiting`/`clientsClaim`, navigationPreload, and a SW→Sentry telemetry bridge.

**The auth-partitioned navigation cache** is the crux: because we cache the logged-in
HTML shell, we had to invent a per-user cache key (SHA-256 of the Supabase `sub` claim)
so User A's cached shell can't be served to User B on a shared device. rhyme solves the
identical risk by simply not caching that shell.

**Manifest:** `standalone`, 4 PNG icons, 3 app shortcuts, absolute-www `start_url` to
dodge an iOS redirect loop, `id: "/"` pinned.

**iOS head:** **17 apple-touch-startup-image entries** covering ~20 device configs
(generated by a Python script), absolute-www URLs to dodge a 307-redirect-to-black-screen
bug. Cold-open smoke overlay (server-rendered, throttled 5min, 18 animated wisps).

**Resilience layer — five independent mechanisms** patching launch/hang failure modes:

| Mechanism | Root cause it patches |
|---|---|
| Stale-chunk recovery (inline head script) | SW served stale HTML referencing a deleted chunk after deploy |
| Hydration watchdog (15s timer + iOS overlay) | Silent hydration hang leaves a frozen black screen |
| ResumeHandler (heap-eviction detect, auth pre-warm, SW update) | iOS evicts PWA JS heap after ~5min background |
| Proxy auth timeout (3s race on `getSession`) | Slow Supabase auth held the document response open |
| ReliabilityBootstrap | Telemetry so we can see which of the above fired |

---

## 4. The core finding

**rhyme is not more advanced than us. It is more advanced by being less ambitious.**

Map our resilience layer against rhyme's architecture and a pattern appears:

| Our failure mode / patch | Does rhyme's model even have this problem? |
|---|---|
| Stale chunk after deploy → cache-nuke recovery | **No.** Network-first means it fetches the live HTML, which references live chunks. Stale shell can't be served while online. |
| Multi-user shell leak → auth-partitioned cache key | **No.** It never caches the authed shell. |
| SW precache install hangs on auth-gated `public/` file | **No.** It precaches 3 public assets. No manifest to poison. |
| "Frozen on resume" from stale cached shell | **Largely no.** App is always network-fetched; resume just refetches. |
| Cold-load white screen → SWR navigation cache | Partially. They get instant launch by caching only the **public** shell; the app loads from network with a fast origin. |

Most of the engineering we are proud of exists to make **offline-first app-shell
caching** survive contact with Next RSC + Supabase auth + iOS WebKit. rhyme declined
that fight. The "works really really well" feeling Dave had is mostly: clean manifest +
fullscreen + a SW that gets out of the way.

This does **not** mean our work was wrong. It means the complexity is the price of one
specific capability: **the app working offline / on flaky networks.** The only question
that matters is whether that capability is worth its price for a cigar humidor app.

---

## 5. Does Ash & Ember actually need offline?

What our offline-first machinery buys, concretely:
- Browsing already-loaded catalog/humidor pages with no signal.
- Logging a burn report offline and replaying it when back online (the IDB outbox).
- Instant repeat launches from cached shell.

Honest read: a humidor/log/community app is **mostly online-by-nature** (the feed,
catalog search, weather, posting all need the network). The genuinely useful offline
case is "log a burn in a cellar/lounge with bad signal, sync later." That is real but
narrow. Instant launch we can get without offline-first (rhyme proves it).

This is the decision Dave owns. The spec lays out three directions so the cost of each
is explicit.

---

## 6. Three directions

### Option A — Adopt rhyme's model (strip to a dumb SW)
Replace Serwist with a small network-first worker. Precache only the public shell
(`/`, manifest, icons, login). **Never cache the authed app.** Keep push notifications.

- **Removes:** auth-partitioned cache, precache-gate CI check, most of the offline
  machinery, and the *reason* for stale-chunk recovery and the resume/refetch dance.
- **Keeps working:** instant-ish launch (cached public shell), push, fast origin.
- **Loses:** true offline browsing, offline burn-report replay.
- **Risk:** medium one-time migration; large permanent reduction in surface area.
- **Net:** biggest reliability + maintainability win. Best fit if offline is "nice to
  have," not core.

### Option B — Keep Serwist, borrow rhyme's safe decisions
Stay offline-capable but adopt rhyme's conservative choices where they cost us nothing:
stop caching the authed navigation shell (drop the auth-partition complexity by not
needing it), simplify icons to one SVG, drop or thin the 17-entry splash set, match
their header posture.

- **Removes:** the multi-user-leak risk class and its cache-key code; splash-script
  maintenance.
- **Keeps:** offline catalog/images, outbox replay, telemetry.
- **Risk:** lower than A; surgical changes, each independently shippable.
- **Net:** middle path. Keeps the offline capability we already built, sheds the
  riskiest part of it (caching the authed shell).

### Option C — Quick wins only (no SW architecture change)
Leave the architecture intact. Pull the cheap, safe parts of rhyme:
1. Manifest: consider `display: fullscreen` + override chain for a more native feel.
2. Icons: collapse to one SVG `sizes:"any"` (kills the PNG set + apple-touch sprawl).
3. Re-examine the 17 splash images — rhyme ships **zero** and feels great; our cold-open
   overlay may already make most of them redundant.
4. Header parity check (we are already close).

- **Risk:** minimal. **Net:** marginal polish, leaves the real complexity in place.

---

## 7. Recommendation

If the goal is "feels like rhyme — fast, native, never breaks," the lever is **Option A
or B, not C.** rhyme's magic is the *absence* of offline-first app-shell caching, and C
leaves that exactly where it is.

Lead recommendation: **Option B as the destination, sequenced so the first step is the
single highest-value change — stop caching the authenticated navigation shell.** That
one change retires our largest open risk (multi-user leak, pending item #2 in
PROJECT_STATE) and a chunk of the resilience layer, while keeping offline catalog/outbox.
If, after that, offline browsing proves unused in telemetry, collapsing the rest toward
Option A becomes a low-risk follow-up.

Whichever direction, the quick wins in C (one-SVG icon, splash-set audit) are safe to do
in parallel.

## 8b. DECISION (2026-06-28) — approved direction

Dave's calls:
- **Offline is nice-to-have, not core.** Most lounges have wifi. Not a critical capability.
- **Splash images: test dropping** the 17-entry `apple-touch-startup-image` set.
- **Cold-smoke animation stays** — users love it; it becomes the launch-feel that
  replaces native splash.
- **One bold PR**, not a sequence.
- **Drop the offline mutation outbox too** — fully commit to the simpler model.

Chosen direction: **Option A, adapted.** Because Ash & Ember is entirely authenticated
(no public `/beta` split like rhyme), "adopt rhyme's model" means: keep caching public
static assets, **stop caching the authenticated app shell.**

### Tooling decision
**Keep Serwist** as the build + registration pipeline; gut its caching config. Hand-rolling
a raw worker like rhyme adds migration risk for no benefit. Bold on behavior, conservative
on plumbing.

### Scope of the PR

Change `app/sw.ts`:

| Resource | Before | After |
|---|---|---|
| App navigation / shell | StaleWhileRevalidate, auth-partitioned | **network-only + cached `/offline` fallback** |
| Supabase REST/Auth, RSC | NetworkOnly | NetworkOnly (unchanged) |
| `/_next/static/*` | CacheFirst + static precache | unchanged (static precache stays) |
| `/_next/image`, same-origin images | StaleWhileRevalidate | unchanged |
| Google Fonts | CacheFirst | unchanged |
| Supabase public storage | StaleWhileRevalidate | unchanged |

**Delete:**
- `authPartitionPlugin` + `navCacheStalePlugin` and their helpers (`extractSubClaim`,
  `authHashForRequest`) — no authed shell to partition.
- The Serwist `fallbacks` block — replaced by a dedicated `/offline` cache the SW
  populates on install, served by a custom network-first navigation handler.
- Offline mutation outbox: IndexedDB `ae-offline-outbox`, `ae-outbox-sync` Background Sync,
  the `sync` listener in `app/sw.ts`, `lib/offline-outbox.ts`, and
  `components/system/OutboxManager.tsx` (with its mount + BurnReport call site).
- 17 `apple-touch-startup-image` entries + the `iosSplash` helper in `app/layout.tsx`.

**Deliberately NOT changed (scope correction, 2026-06-28):**
- The static-only precache (`...self.__SW_MANIFEST`) stays. It was already trimmed to
  hashed `/_next/static` (immutable, always-200) on 2026-05-30; it is not a source of
  pain, and it backs the `SW_VERSION` update banner. Dropping it would be gratuitous.
- The `sw-precache-check` CI gate stays — it still guards the unchanged precache
  regression class (an auth-gated route sneaking back into the manifest).
- `serwist.config.mjs`, push notifications, telemetry, cold-smoke overlay, hydration
  watchdog, ResumeHandler, proxy auth timeout — all untouched.

**Activate handler:** must purge **all** prior Serwist caches on upgrade (bump cache
version) so no stale authenticated HTML survives the switch. This is the critical
migration step.

**Keep (untouched):**
- Cold-smoke overlay (launch feel).
- Push notifications.
- Hydration watchdog, ResumeHandler (heap-eviction + auth pre-warm + SW update),
  proxy auth timeout, ReliabilityBootstrap/Sentry telemetry.
- Stale-chunk recovery (tiny, harmless insurance; also now near-impossible to trigger).

### Cold-launch note (corrected 2026-06-28 after final review)
Removing the StaleWhileRevalidate navigation cache does reintroduce the cold-launch
TTFB vector that cache was originally added to kill: a cold iOS relaunch hits `proxy.ts`
on a cold connection before the first HTML byte arrives. The earlier claim that "the
cold-smoke overlay covers the gap" is not quite right — the overlay is server-rendered,
so it ships *inside* the HTML body that hasn't arrived yet and cannot paint during the
pre-TTFB window. What actually bounds the regression is (a) the existing 3s proxy
auth-timeout race, and (b) navigation preload, which the network-first handler now
explicitly consumes (`event.preloadResponse`) so the navigation fetch starts in parallel
with SW boot. This is a Dave-approved tradeoff: offline is nice-to-have, and the
multi-user-leak + stale-shell classes are retired in exchange. The window is bounded, not
eliminated; watch Speed Insights cold-launch LCP after deploy.

### Risk + rollback
Touches `app/sw.ts`, the file behind prior black-screen incidents. Steady state after is
**safer** (network-first SW cannot serve stale or wrong-user shells). The risk window is
the migration launch (worker swap + cache purge). Mitigations: `/sw.js` is
`max-age=0, must-revalidate` so a bad worker is replaced on next launch (not stuck);
telemetry stays live; rollback = revert the single PR. CACHE_NAME bump guarantees old
caches are purged.

### Success criteria
- Installed PWA launches and navigates correctly online after the worker swap.
- No authed HTML is read from cache (verify: navigation requests hit network; on offline,
  `/offline` shown).
- Old Serwist caches (esp. `navigations`/auth-partitioned) are gone after activate.
- Push notifications still deliver and route taps.
- Cold-smoke overlay still plays on cold open (throttled).
- Splash images removed; launch still feels covered by cold-smoke.
- Telemetry shows no rise in `sw_lifecycle` / `ios_webkit` reliability events post-deploy.
